# 《苍穹外卖》12天渐进式开发计划

本计划基于 CSDN 实战笔记与黑马程序员全套视频制定，旨在帮助开发者从零构建企业级外卖平台。计划涵盖前后端开发、数据库设计、测试与代码审查，确保每一行代码都经得起推敲。

## 📅 第1阶段：基础架构与员工管理（Day 1-3）

### Day 1: 环境搭建与基础登录
**目标**：完成后端工程搭建、Nginx 反向代理配置及 Swagger 接口文档集成。
1.  **功能模块**：
    -   管理端登录接口 (`/admin/employee/login`)
    -   退出登录接口 (`/admin/employee/logout`)
2.  **前端页面**：
    -   配置 Nginx 反向代理 (解决跨域问题)
    -   管理端登录页面 (`login.html`)
3.  **数据库设计**：
    -   表：`employee` (员工表)
    -   SQL脚本：`sky_take_out.sql` (初始化建表语句)
4.  **测试用例**：
    -   测试登录成功/失败场景（密码错误、账号锁定）。
    -   Swagger 接口文档访问测试。
5.  **Code Review**：
    -   MD5 加密实现细节。
    -   JWT 令牌生成与解析逻辑。
    -   全局异常处理器 (`GlobalExceptionHandler`) 配置。
6.  **产出物**：
    -   分支：`feature/init-login`
    -   文档：`doc/api_login.md`

### Day 2: 员工管理与分页查询
**目标**：实现员工的增删改查及分页功能，引入 ThreadLocal 优化用户上下文。
1.  **功能模块**：
    -   新增员工 (`POST /admin/employee`)
    -   员工分页查询 (`GET /admin/employee/page`)
    -   启用/禁用员工账号 (`POST /admin/employee/status/{status}`)
    -   编辑员工信息 (`PUT /admin/employee`)
2.  **前端页面**：
    -   员工列表页 (`employee/list.html`)
    -   新增/编辑员工弹窗。
3.  **数据库设计**：
    -   `employee` 表增加索引优化查询。
4.  **测试用例**：
    -   分页参数边界测试（page=1, pageSize=10）。
    -   账号唯一性约束测试（重复用户名）。
5.  **Code Review**：
    -   **易错点**：日期格式转换（Jackson 配置）。
    -   **扩展性**：ThreadLocal 存储当前登录用户 ID，用于后续字段自动填充。
6.  **产出物**：
    -   分支：`feature/employee-crud`

### Day 3: 分类管理与公共字段自动填充
**目标**：完成分类管理，并使用 AOP 实现公共字段（create_time, update_time 等）自动填充。
1.  **功能模块**：
    -   分类增删改查 (`/admin/category`)
    -   公共字段自动填充切面 (`AutoFillAspect`)
2.  **前端页面**：
    -   分类管理页 (`category/list.html`)
3.  **数据库设计**：
    -   表：`category` (分类表)
4.  **测试用例**：
    -   删除分类时关联检查（若分类下有菜品则禁止删除）。
    -   AOP 切面是否正确拦截并填充字段。
5.  **Code Review**：
    -   **设计意图**：AOP 减少重复代码（DRY原则）。
    -   **性能**：反射机制对性能的微小影响 vs 开发效率提升。
6.  **产出物**：
    -   分支：`feature/category-aop`

## 📅 第2阶段：核心业务开发（Day 4-6）

### Day 4: 文件上传与菜品管理（上）
**目标**：集成 OSS/MinIO 文件上传服务，完成菜品新增功能。
1.  **功能模块**：
    -   通用文件上传接口 (`/admin/common/upload`)
    -   新增菜品接口 (`POST /admin/dish`)
2.  **前端页面**：
    -   菜品管理页
    -   图片上传组件集成
3.  **数据库设计**：
    -   表：`dish` (菜品表), `dish_flavor` (口味表)
    -   关系：一对多（菜品-口味）
4.  **测试用例**：
    -   图片格式与大小限制测试。
    -   菜品关联口味的数据一致性测试。
5.  **Code Review**：
    -   **易错点**：事务控制 (`@Transactional`)，确保菜品和口味同时保存成功。
    -   **配置**：OSS 密钥安全性（配置在 yml 或环境变量）。
6.  **产出物**：
    -   分支：`feature/dish-add`

### Day 5: 菜品管理（下）与套餐管理（上）
**目标**：完成菜品修改、删除、起停售，及套餐的新增功能。
1.  **功能模块**：
    -   批量删除菜品 (`DELETE /admin/dish`)
    -   根据 ID 查询菜品（回显）
    -   新增套餐 (`POST /admin/setmeal`)
2.  **前端页面**：
    -   菜品修改页
    -   套餐管理页
3.  **数据库设计**：
    -   表：`setmeal` (套餐表), `setmeal_dish` (套餐菜品关系表)
4.  **测试用例**：
    -   删除包含起售中菜品的异常测试。
    -   修改菜品时同步更新口味数据（先删后增策略）。
5.  **Code Review**：
    -   **性能**：批量删除时的 SQL 优化 (`IN` 查询)。
    -   **逻辑**：套餐价格计算逻辑校验。
6.  **产出物**：
    -   分支：`feature/dish-update-setmeal`

### Day 6: 套餐管理（下）与店铺营业状态
**目标**：完善套餐业务，使用 Redis 管理店铺营业状态。
1.  **功能模块**：
    -   套餐分页查询、修改、删除、起停售
    -   店铺营业状态设置/查询 (`/admin/shop/status`, `/user/shop/status`)
2.  **前端页面**：
    -   套餐列表操作栏
    -   管理端/用户端营业状态展示
3.  **数据库设计**：
    -   Redis Key: `SHOP_STATUS`
4.  **测试用例**：
    -   Redis 连接与读写测试。
    -   套餐起售时检查内部菜品是否全部起售。
5.  **Code Review**：
    -   **扩展性**：RedisTemplate 序列化配置。
    -   **设计**：状态位使用 Redis 而非 DB 的原因（高频访问）。
6.  **产出物**：
    -   分支：`feature/setmeal-shop-status`

## 📅 第3阶段：C端用户与订单核心（Day 7-9）

### Day 7: 微信登录与商品浏览
**目标**：实现 C 端用户微信登录，完成商品列表展示。
1.  **功能模块**：
    -   用户微信登录 (`/user/user/login`)
    -   查询分类及分类下菜品/套餐
2.  **前端页面**：
    -   微信小程序登录授权
    -   首页商品瀑布流
3.  **数据库设计**：
    -   表：`user` (用户表，存 openid)
4.  **测试用例**：
    -   HttpClient 调用微信接口模拟测试。
    -   未登录拦截器测试。
5.  **Code Review**：
    -   **安全**：JWT 令牌在 C 端的存储与传递。
    -   **逻辑**：HttpClient 发送 GET/POST 请求封装。
6.  **产出物**：
    -   分支：`feature/wechat-login`

### Day 8: 缓存优化与购物车
**目标**：引入 Spring Cache 优化查询性能，实现购物车增删改查。
1.  **功能模块**：
    -   缓存菜品/套餐数据 (`@Cacheable`, `@CacheEvict`)
    -   添加/查看/清空购物车 (`/user/shoppingCart`)
2.  **前端页面**：
    -   购物车浮层/页面
3.  **数据库设计**：
    -   表：`shopping_cart` (购物车表)
    -   Redis Key: `dish_*`, `setmeal_*`
4.  **测试用例**：
    -   缓存一致性测试（修改菜品后缓存是否清理）。
    -   购物车添加逻辑（同一规格累加数量）。
5.  **Code Review**：
    -   **性能**：Redis 缓存穿透/雪崩预防策略简述。
    -   **设计**：购物车数据存 DB vs Redis 的权衡（本项目存 DB）。
6.  **产出物**：
    -   分支：`feature/cache-cart`

### Day 9: 地址簿与用户下单
**目标**：完成地址簿管理，实现用户下单核心逻辑。
1.  **功能模块**：
    -   地址簿增删改查 (`/user/addressBook`)
    -   用户下单 (`POST /user/order/submit`)
2.  **前端页面**：
    -   地址选择页
    -   订单确认页
3.  **数据库设计**：
    -   表：`address_book`, `orders` (订单表), `order_detail` (订单明细表)
4.  **测试用例**：
    -   库存不足下单失败测试。
    -   事务回滚测试（插入明细失败回滚订单）。
5.  **Code Review**：
    -   **核心**：订单号生成策略。
    -   **易错点**：金额计算精度问题（使用 BigDecimal）。
6.  **产出物**：
    -   分支：`feature/order-submit`

## 📅 第4阶段：订单处理与数据统计（Day 10-12）

### Day 10: 订单支付与状态流转
**目标**：模拟微信支付流程，实现订单状态流转（支付成功 -> 接单 -> 派送）。
1.  **功能模块**：
    -   订单支付 (`/user/order/payment`) - *可Mock*
    -   查询历史订单 (`/user/order/historyOrders`)
    -   商家接单/拒单/派送 (`/admin/order`)
2.  **前端页面**：
    -   支付成功页
    -   订单详情页（状态时间轴）
3.  **数据库设计**：
    -   `orders` 表状态字段更新逻辑。
4.  **测试用例**：
    -   支付回调逻辑测试。
    -   状态机流转合法性校验（如：待支付不能直接变已完成）。
5.  **Code Review**：
    -   **设计**：支付回调接口的幂等性处理。
6.  **产出物**：
    -   分支：`feature/order-status`

### Day 11: 订单超时处理与来单提醒
**目标**：使用 Spring Task 处理超时订单，WebSocket 实现商家来单/催单语音播报。
1.  **功能模块**：
    -   定时任务：处理超时未支付/一直派送中的订单 (`Spring Task`)
    -   来单提醒/用户催单 (`WebSocket`)
2.  **前端页面**：
    -   管理端消息弹窗与语音播报。
3.  **数据库设计**：
    -   无新增表，主要涉及 Cron 表达式配置。
4.  **测试用例**：
    -   修改系统时间模拟超时场景。
    -   多客户端 WebSocket 消息推送测试。
5.  **Code Review**：
    -   **性能**：定时任务对数据库的压力评估。
    -   **扩展性**：WebSocket 消息格式定义。
6.  **产出物**：
    -   分支：`feature/task-websocket`

### Day 12: 数据统计与工作台
**目标**：基于 Apache ECharts 实现营业额、用户、订单统计，完善工作台数据展示。
1.  **功能模块**：
    -   营业额统计 (`/admin/report/turnoverStatistics`)
    -   用户/订单/销量Top10统计
    -   Excel 导出报表 (`Apache POI`)
2.  **前端页面**：
    -   数据报表页 (ECharts 图表)
    -   工作台概览
3.  **数据库设计**：
    -   复杂聚合查询 SQL (`SUM`, `COUNT`, `DATE_FORMAT`)
4.  **测试用例**：
    -   统计数据准确性校验。
    -   大范围日期查询性能测试。
5.  **Code Review**：
    -   **设计**：代码层组装数据 vs SQL 层聚合数据的选择。
    -   **总结**：项目整体架构回顾与优化点梳理。
6.  **产出物**：
    -   分支：`feature/report-workspace`
    -   文档：`doc/final_report.md`
