# 苍穹外卖项目汇报稿

更新日期：2026-03-27

这份文档适合 3 类场景：

- 给老师或答辩老师做项目汇报

- 给同学或接手人做项目交接

## 一、项目一句话概述

苍穹外卖项目是一套前后端联动的餐饮点餐与运营管理系统。  
本次交付在原有 Spring Boot 后端基础上，新增了 `sky-web` 双入口前端，实现了：

- 顾客端点餐、下单、支付、催单、地址管理
- 商家/管理员后台的工作台、订单流转、菜品套餐管理、店铺状态、报表统计
- 双入口可访问、真实接口可联调、端到端验收脚本可复跑

## 二、项目目标

这次项目的目标不是只做接口，而是把它交付成一套“能演示、能联调、能验证”的完整系统。

具体目标有 3 个：

1. 顾客能够完成一次完整点餐流程
2. 商家或管理员能够在后台完成接单与经营管理
3. 项目能够通过脚本化方式证明“当前版本是可运行、可交付的”

## 三、系统结构

### 1. 后端

- 技术栈：Spring Boot + MyBatis + Redis + Knife4j
- 主要模块：
  - 员工管理
  - 分类管理
  - 菜品管理
  - 套餐管理
  - 店铺状态
  - 订单管理
  - 工作台统计
  - 报表导出
  - WebSocket 来单提醒

### 2. 前端

- 技术栈：React + Vite + TypeScript + React Router + TanStack Query + ECharts
- 双入口设计：
  - `/customer/*` 顾客端
  - `/console/*` 后台端

### 3. 部署与代理

- 后端：`http://127.0.0.1:8080`
- Nginx：`http://127.0.0.1`
- 顾客端入口：`http://127.0.0.1/customer/`
- 后台端入口：`http://127.0.0.1/console/`

## 七、如何证明这版是“可交付”的

### 1. 后端验证

```powershell
./mvnw -q -pl sky-server -am test
./mvnw -q -pl sky-server -am -DskipTests compile
```

### 2. 前端验证

```powershell
cd .\sky-web
npm run lint
npm run build
```

### 3. 双入口端到端验收

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\sky_web_end_to_end_acceptance.ps1
```

当前已验证通过的关键结果：

- `customerEntryOk = true`
- `consoleEntryOk = true`
- `docEntryOk = true`
- `finalStatus = 5`
- `finalPayStatus = 1`

这代表：

- 顾客端入口正常
- 后台端入口正常
- 接口文档正常
- 订单最终能走到“已完成”
- 支付状态最终能走到“已支付”

## 八、项目当前边界

这版项目已经能演示和联调交付，但仍然有边界需要说明：

- 顾客登录仍然是 mock 登录，不是正式微信登录
- 支付仍然是 mock 支付，不是正式支付通道
- 后台权限分层主要由前端视图控制，没有单独新增 RBAC
- 部分历史文案和编码仍可继续做细节统一，但不影响当前运行

## 十、一分钟口播版

本项目是一套完整的外卖点餐与运营管理系统。后端基于 Spring Boot，前端新增了 React 双入口，分别面向顾客端和后台端。顾客端已经能够完成浏览菜品、购物车、地址簿、提交订单、支付、催单等完整流程；后台端能够完成工作台查看、订单处理、菜品套餐管理、店铺状态切换和报表查看。我们还补充了端到端验收脚本，能够实际证明系统从顾客下单到后台完成订单的整条链路已经跑通，因此这版项目已经达到可演示、可联调、可交付的状态。

## 十一、三分钟口播版

本项目的核心目标，是把“后端接口项目”进一步做成“真正可运行、可展示、可验证”的完整系统。  
在后端方面，我们已经完成了员工、分类、菜品、套餐、订单、工作台、报表、WebSocket 提醒等模块；在前端方面，我们新增了一个 `sky-web` 工程，使用 React + Vite + TypeScript，做成了顾客端和后台端双入口。

顾客端主要解决的是点餐闭环：用户进入后会自动完成 mock 登录，然后可以浏览菜品和套餐、加入购物车、维护地址簿、提交订单，再到订单中心完成支付、催单、取消和再来一单。  
后台端主要解决的是运营闭环：管理员登录后可以看到工作台、订单中心、分类管理、菜品管理、套餐管理、店铺状态、报表中心和员工管理。整个后台不仅能看，还能真实调用后端接口完成接单、派送和完成订单。

为了证明项目不是“只能看页面”，我们还补充了端到端验收脚本。这份脚本会从访问顾客端和后台端开始，自动完成顾客登录、加菜、提交订单、支付，然后再由后台完成接单、派送和完成，最后输出订单状态和支付状态。当前结果已经验证通过，说明这版系统已经具备完整演示和联调交付能力。

## 十二、答辩常见问法参考

### 1. 你们项目最大的改进是什么？

最大的改进是把原本偏后端练习型项目，做成了真正的前后端联动交付版，尤其是新增了双入口前端和端到端验收脚本。

### 2. 你们怎么证明这个系统是真的可用？

我们不是只展示界面，而是通过：

- 后端测试与编译
- 前端 lint 与 build
- 双入口端到端验收脚本

来证明这版项目可以完整运行。

### 3. 为什么顾客端和后台端要分成双入口？

因为两类用户的使用场景不同。顾客端强调移动端点餐体验，后台端强调桌面端运营和管理效率，所以拆成双入口更符合实际产品形态。

### 4. 目前还没有做的是什么？

正式微信登录、正式支付、后端 RBAC 细化权限，以及更深入的运营分析能力，这些属于后续迭代方向。
