# 前端源码替换评估结论（2026-04-07）

## 结论摘要

当前更推荐的选择是：

1. 保留现有 `E:\sky\sky-web` 作为正式前端主线。
2. 将压缩包里的 Vue 管理后台作为“旧后台候选/参考实现”保留在候选目录，不直接替换。
3. 如果后续确实要用这份 Vue 源码，也只建议作为独立后台试运行，不建议直接覆盖当前 `sky-web`。

本次没有执行正式替换，原因不是“还没试完”，而是已经确认这份候选源码与当前正式前端的结构目标不一致，直接替换会让现有双入口能力倒退。

## 当前环境事实

- 当前正式前端目录：`E:\sky\sky-web`
- 当前正式前端技术栈：`React + Vite + TypeScript`
- 当前正式前端对外入口：
  - `/customer/*`
  - `/console/*`
- 当前前端已适配：
  - 顾客端 `authentication`
  - 后台端 `token`
  - `Nginx + /api/user/* + /api/admin/* + /ws/admin/orders`

压缩包解压后的实际候选项目为：

- 源包：`E:\前端源码\苍穹外卖前端源码.rar`
- 候选目录：`E:\sky\sky-web-candidate\苍穹外卖前端源码\苍穹外卖前端源码\project-sky-admin-vue-ts`

候选项目实际技术栈为：

- `Vue 2`
- `Vue CLI 3`
- `TypeScript 3.6`
- `Element UI`
- `Webpack 4`

## 已发现的问题

### 1. 结构不匹配

这是本次最重要的问题。

- 当前正式前端是“双入口”：
  - 顾客端
  - 后台端
- 压缩包里只有“后台管理项目”，没有顾客端对应实现。

这意味着它不能直接替换整个 `sky-web`，否则会丢掉当前顾客端页面和链路。

### 2. 候选项目是老技术栈

候选项目依赖明显偏旧：

- Vue 2
- Vue CLI 3
- Webpack 4
- TypeScript 3.6

而当前机器环境是：

- Node `v24.14.0`
- npm `11.9.0`
- yarn `1.22.22`

这类老项目在 Node 24 上存在明显兼容风险，尤其是构建工具链和部分老依赖。

### 3. 依赖安装已实际失败

我已经在候选目录尝试安装依赖，遇到的第一个硬阻塞是：

- `yarn install` 失败
- 失败原因：锁文件中的淘宝源地址证书过期

具体表现：

- `yarn.lock` 和 `package-lock.json` 都写死了 `registry.npm.taobao.org`
- 安装时报 `SSL Error: CERT_HAS_EXPIRED`

这说明即使不考虑代码结构问题，这份候选项目当前也不能直接在这台机器上完成安装。

### 4. 候选项目的代理与当前后端约定不一致

候选项目的开发代理是：

- `/api/* -> http://localhost:8080/admin`

当前正式前端的约定是：

- `/api/user/* -> /user/*`
- `/api/admin/* -> /admin/*`
- `/ws/admin/orders`

这意味着：

- 候选项目只覆盖后台管理接口
- 没有顾客端接口分流
- 也没有当前双入口项目使用的前后端代理规范

### 5. WebSocket 约定不一致

候选项目环境变量里写的是：

- `VUE_APP_SOCKET_URL = 'ws://localhost:8080/ws/'`

而当前正式项目使用的是：

- `ws://127.0.0.1:8080/ws/admin/orders`

如果直接替换，不做适配的话，实时提醒链路大概率不能直接接上当前后端。

### 6. 构建与部署方式不一致

当前正式前端：

- 构建输出到 `E:\sky\nginx\html\sky-web`
- Nginx 直接提供 `/customer/` 和 `/console/`

候选 Vue 项目当前配置：

- `publicPath` 仍是模板式配置
- 开发端口写成 `8888`
- 构建目标并没有对齐当前 Nginx 双入口结构

也就是说，即使它能装起来，也还不能无缝替换当前部署方式。

## 推荐选择（按优先级排序）

### 选择 A：保留当前 React 双入口，候选 Vue 项目只做参考

这是最推荐的选择。

适用场景：

- 你要保证当前顾客端 + 后台端继续可运行
- 你不想因为替换而丢掉双入口结构
- 你更重视“当前项目稳定交付”

原因：

- 当前 `sky-web` 已经是完整双入口
- 与现有后端接口、Nginx、鉴权头、WebSocket 都对齐
- 替换成候选 Vue 项目会造成功能倒退

### 选择 B：候选 Vue 项目只作为“独立后台候选”并行验证

这是次推荐选择。

适用场景：

- 你确实想试用这份 Vue 管理后台
- 但不打算替换顾客端

建议做法：

- 不覆盖 `E:\sky\sky-web`
- 将候选项目单独跑在新目录
- 仅作为后台管理页试运行
- 需要额外适配：
  - 官方 npm registry
  - Node 版本降级到 14/16 LTS
  - 代理地址与 WebSocket 地址
  - 构建输出目录

### 选择 C：直接替换当前 `sky-web`

这是最不推荐的选择。

不推荐原因：

- 会失去当前顾客端能力
- 安装依赖还没过
- 技术栈明显老旧
- 当前代理和部署结构不兼容
- 替换成本远高于收益

## 如果坚持要用候选 Vue 项目，建议的最小改造项

若后续要继续推进这份 Vue 候选项目，优先顺序建议如下：

1. 删除或重建锁文件，改用官方 npm registry
2. 切换 Node 到 `14.x` 或 `16.x` LTS 后再安装依赖
3. 修正开发代理：
   - `/api/admin/* -> /admin/*`
4. 修正 WebSocket 地址到：
   - `/ws/admin/orders`
5. 明确构建产物如何部署到当前 Nginx
6. 明确它只替代后台端，不替代顾客端
